# UX_UI_PRINCIPLES.md

## 1. Основная концепция

Интерфейс Marketing Intelligence должен сочетать:

- простоту для начинающего пользователя;
- высокую функциональность для опытного пользователя;
- минимальную визуальную нагрузку;
- доступ ко всем необходимым действиям без изучения документации.

Базовый пользователь должен полноценно работать с продуктом, используя только явно видимые основные элементы.

Расширенные функции не должны создавать ощущение сложности, перегружать экран или конкурировать за внимание с основными действиями.

## 2. Прогрессивное раскрытие

В исходном состоянии интерфейс показывает:

- основное действие;
- основные данные;
- основные фильтры;
- основную навигацию;
- текущий результат;
- понятный следующий шаг.

Дополнительные действия появляются:

- после нажатия на дополнительную область элемента;
- после раскрытия меню;
- после открытия расширенных параметров;
- при взаимодействии с конкретной строкой, карточкой, графиком или фильтром;
- в контексте текущей задачи.

Расширенные функции не должны постоянно занимать место, если они не нужны большинству пользователей.

## 3. Один интерфейс для разных уровней

Не должно быть обязательного разделения на режимы:

- простой;
- профессиональный;
- экспертный.

Все пользователи работают в одном интерфейсе.

Начинающий пользователь использует основное действие. Опытный пользователь раскрывает дополнительные возможности того же элемента.

## 4. Основные действия

Каждый экран должен содержать одно визуально приоритетное действие.

Примеры:

- «Добавить сайт»;
- «Запустить проверку»;
- «Применить фильтры»;
- «Сравнить изменения»;
- «Экспортировать отчёт».

Основная часть элемента всегда выполняет наиболее частое и безопасное действие.

## 5. Составные кнопки

Составная кнопка может содержать:

- основную область с базовым действием;
- дополнительную область, раскрывающую список других действий.

Обе области могут:

- иметь единый цвет;
- восприниматься как один элемент;
- не иметь контрастного разделителя;
- иметь общий внешний контур.

При этом:

- области должны иметь разные зоны нажатия;
- основная зона всегда выполняет главное действие;
- дополнительная зона никогда не запускает главное действие;
- при наведении и фокусе должна быть ненавязчивая обратная связь;
- меню должно закрываться по клику вне элемента и клавишей Escape;
- управление должно работать с клавиатуры;
- опасные действия не должны становиться основными.

## 6. Обнаруживаемость

Дополнительные функции не должны привлекать лишнее внимание, но пользователь, который их ищет, должен суметь их обнаружить.

Допускаются:

- небольшой значок раскрытия;
- изменение состояния при наведении;
- подсказка;
- контекстное меню;
- ссылка «Дополнительные параметры»;
- раскрытие по нажатию на строку или карточку.

Недопустимо делать дополнительную функцию полностью невидимой и непредсказуемой.

## 7. Формы

В исходном состоянии форма содержит только поля, необходимые для базовой операции.

Пример формы добавления сайта:

Базовые поля:

- название;
- URL.

Расширенные поля:

- ограничение количества страниц;
- глубина обхода;
- задержка;
- исключаемые URL;
- правила обработки параметров;
- расписание.

Безопасные значения по умолчанию должны позволять выполнить действие без открытия расширенных параметров.

## 8. Фильтры

Сразу показываются наиболее востребованные фильтры.

Редкие фильтры раскрываются дополнительно.

Основной сценарий:

1. выбрать основные параметры;
2. нажать «Применить фильтры»;
3. получить результат.

Расширенный сценарий может включать:

- сохранение набора;
- сложные условия;
- исключения;
- сохранение представления;
- экспорт результата.

## 9. Таблицы и списки

В таблице показываются только наиболее важные столбцы.

Дополнительные данные открываются через:

- раскрытие строки;
- боковую панель;
- настройку столбцов;
- контекстное меню;
- подробную карточку.

Каждая строка должна иметь одно очевидное основное действие.

## 10. Карточки

Карточка в базовом состоянии показывает:

- название;
- текущее состояние;
- главный показатель;
- последнее важное изменение;
- основное действие.

Подробные параметры и технические данные раскрываются по запросу.

## 11. Обратная связь

Интерфейс обязан показывать:

- наведение;
- фокус;
- нажатие;
- загрузку;
- выполнение операции;
- успешное завершение;
- ошибку;
- недоступность действия;
- несохранённые изменения.

Пользователь должен понимать, что произошло и что делать дальше.

## 12. Доступность

Все интерактивные элементы должны:

- поддерживать клавиатуру;
- иметь видимый фокус;
- иметь понятное текстовое название;
- обладать достаточной областью нажатия;
- не зависеть только от цвета;
- сохранять читаемость при масштабировании;
- не требовать сверхточного наведения мышью.

Минимализм не должен ухудшать доступность.

### 12.1. Отображение изменённых данных

При сравнении старого и нового значения интерфейс показывает стороны в одном обязательном порядке:

1. `Стало` — текущее значение сверху с зелёным визуальным акцентом.
2. Между значениями — явный вертикальный разделитель.
3. `Было` — предыдущее значение снизу с красным визуальным акцентом.

Подписи `Стало` и `Было` отображаются текстом и не заменяются цветом. Зелёный и красный акценты должны сохранять достаточный контраст, поэтому различие остаётся понятным при нарушениях цветовосприятия и в монохромном представлении.

Правило применяется ко всем сравниваемым данным, включая метаданные, текст и внутренние ссылки. Если одна сторона отсутствует, интерфейс явно обозначает отсутствие значения и не меняет порядок сторон.

## 13. Опасные действия

К опасным действиям относятся:

- удаление сайта;
- удаление истории;
- очистка базы;
- массовое изменение;
- сброс настроек;
- остановка критической операции.

Они должны:

- быть визуально отделены внутри меню;
- требовать подтверждения;
- ясно описывать последствия;
- по возможности иметь отмену или восстановление.

Основная часть составной кнопки никогда не выполняет опасное действие.

## 14. Единообразие

Один тип взаимодействия должен работать одинаково во всём продукте.

Необходимо стандартизировать:

- составные кнопки;
- раскрывающиеся меню;
- расширенные параметры;
- действия строк;
- подтверждения;
- загрузку;
- ошибки;
- обозначение фактов и гипотез.

Пользователь должен изучить принцип один раз.

## 15. Проверка нового элемента

Перед добавлением элемента нужно ответить:

1. Понятно ли основное действие без инструкции?
2. Доступна ли базовая функция одним очевидным способом?
3. Не конкурируют ли дополнительные действия с основным?
4. Может ли опытный пользователь найти расширенные возможности?
5. Не скрыта ли важная функция?
6. Есть ли понятная обратная связь?
7. Работает ли управление с клавиатуры?
8. Исключено ли случайное опасное действие?
9. Соответствует ли поведение другим экранам?
10. Можно ли убрать что-либо без потери основного сценария?

## 16. Формула интерфейса

**Простое базовое состояние → очевидное основное действие → контекстное раскрытие → расширенная функциональность без визуальной перегрузки.**

Цель — не сокращать количество функций, а организовать их так, чтобы пользователь видел только необходимый уровень сложности.

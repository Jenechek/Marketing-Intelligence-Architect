# ROADMAP.md — Дорожная карта Marketing Intelligence

## Общий принцип

Разработка идёт небольшими завершёнными этапами.

Каждый этап должен давать проверяемый результат в интерфейсе. Следующий этап начинается только после проверки предыдущего.

## Правило движения по плану

Этапы и задачи выполняются последовательно.

Переход к следующей задаче разрешён только после успешного выполнения, проверки и документирования текущей задачи.

Если порядок работ меняется, изменение явно фиксируется в `TASKS.md` и, при необходимости, в `ROADMAP.md`.

## Этап 0. Подготовка проекта — завершён 2026-07-14

Цель: создать минимальную рабочую основу.

До начала изменения кода должны быть подготовлены и согласованы бизнес-, функциональные и нефункциональные требования в `REQUIREMENTS.md`.

Результат:

- структура проекта;
- локальное Python-окружение;
- FastAPI-приложение;
- SQLite;
- одна стартовая страница;
- один автоматический тест;
- инструкция запуска на Windows;
- Git-репозиторий;
- базовая документация;
- согласованный `REQUIREMENTS.md` в Google Drive и GitHub.

Пользовательская приёмка пройдена: приложение проверено в Яндекс Браузере, Google Chrome и Mozilla Firefox, создание локальной базы и журнала подтверждено, автоматические тесты прошли с отдельным временным каталогом.

## Этап 1. Управление сайтами — завершён 2026-07-14

Цель: пользователь может добавить сайт в систему.

Результат:

- список сайтов;
- форма добавления сайта;
- редактирование сайта;
- удаление сайта с подтверждением;
- проверка URL;
- понятные сообщения об ошибках;
- хранение данных в SQLite.

Пользовательская приёмка пройдена: добавление, редактирование, проверка URL и окончательное удаление сайта с отдельным защищённым подтверждением проверены в интерфейсе; сохранение и удаление данных подтверждены после перезапуска с одной SQLite.

## Этап 2. Первый обход сайта — завершён 2026-07-16

Цель: система умеет безопасно получать страницы одного сайта.

Результат:

- ручной запуск обхода;
- ограничение числа страниц;
- ограничение глубины;
- задержка между запросами;
- обработка таймаутов и ошибок;
- сохранение статуса обхода;
- отображение прогресса;
- журнал ошибок.

Независимая приёмка пройдена: последовательный loopback-обход подтвердил безопасные ограничения, изменяемые настройки, живой прогресс, сохранение результата и понятный журнал HTTP-ошибки после частичного завершения и перезапуска приложения; 102 автоматических теста прошли.

## Этап 3. Сохранение снимка сайта

Цель: система сохраняет состояние страниц на момент проверки.

Результат:

- URL;
- код ответа;
- Title;
- Description;
- H1;
- текст страницы;
- внутренние ссылки;
- дата проверки;
- контрольная сумма содержимого;
- сохранение частично завершённого обхода без потери предыдущих данных.

## Этап 4. Повторный обход и поиск изменений

Цель: система сравнивает два снимка.

Результат:

- новые страницы;
- удалённые страницы;
- изменённые страницы;
- изменение Title;
- изменение Description;
- изменение H1;
- изменение текста;
- изменение внутренних ссылок;
- уровень важности изменения;
- фильтрация событий.

## Этап 5. История изменений конкурентов

Цель: пользователь получает удобную хронологию.

Результат:

- лента изменений;
- фильтры по сайту, дате и типу события;
- просмотр старой и новой версии;
- краткое объяснение изменения;
- отметка «просмотрено»;
- экспорт данных в локальный файл.

## Этап 6. Карта структуры сайта

Цель: визуализировать архитектуру сайта.

Результат:

- дерево страниц;
- граф внутренних ссылок;
- глубина страницы;
- число входящих и исходящих ссылок;
- битые ссылки;
- страницы с недостаточной внутренней связностью;
- фильтры;
- раскрытие подробностей по странице.

## Этап 7. Планировщик проверок

Цель: автоматизировать локальные проверки.

Результат:

- расписание для каждого сайта;
- безопасные значения по умолчанию;
- запуск только при работающем приложении или локальной службе;
- журнал запусков;
- уведомление о сбоях;
- ручной повторный запуск.

## Этап 8. Импорт данных Search Console и аналитики

Цель: использовать данные без обязательных платных API.

Результат:

- импорт CSV;
- проверка структуры файла;
- сопоставление страниц и дат;
- хранение показов, кликов, позиций и трафика;
- отчёт об ошибках импорта;
- повторный импорт без неконтролируемых дублей.

## Этап 9. Ограниченный мониторинг поисковой выдачи

Цель: создать бесплатный и безопасный базовый режим.

Возможные варианты:

- ручной импорт результатов;
- официальный бесплатный источник;
- ограниченная проверка без обхода защит;
- локальное хранение истории.

Результат:

- список запросов;
- история позиций;
- изменение состава выдачи;
- появление новых доменов;
- изменение URL и сниппетов;
- ясное отображение ограничений данных.

## Этап 10. Анализ причин падения трафика

Цель: связывать события из разных источников.

Результат:

- временная шкала событий;
- сопоставление падения трафика с изменениями сайта;
- сопоставление с изменениями позиций;
- сопоставление с действиями конкурентов;
- формирование гипотез;
- степень уверенности;
- список данных, которых не хватает;
- рекомендации по проверке.

## Этап 11. Улучшение UX/UI

Работа над UX ведётся на каждом этапе, но здесь выполняется системная проверка:

- единообразие компонентов;
- прогрессивное раскрытие;
- доступность с клавиатуры;
- понятность ошибок;
- отсутствие визуальной перегрузки;
- скорость выполнения основных сценариев;
- проверка интерфейса пользователем без технических знаний.

## Будущие этапы, не входящие в MVP

- локальная AI-модель;
- установка на собственный сервер;
- многопользовательский режим;
- роли и права;
- публичная облачная версия;
- дополнительные источники данных;
- API продукта.

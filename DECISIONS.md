# DECISIONS.md — Журнал решений

Этот файл фиксирует принятые продуктовые и технические решения.

## DEC-001. Формат продукта

**Решение:** первая версия является локальным веб-приложением.

**Причина:** продукт должен работать без обязательного платного хостинга и открываться через обычный браузер.

**Последствия:**

- пользователь запускает приложение на своём компьютере;
- доступ через интернет не гарантируется;
- обязательное целевое развёртывание на корпоративном сервере выполняется позднее по DEC-022.

## DEC-002. Финансовая модель разработки

**Решение:** основные функции должны работать без платных ресурсов.

**Запрещённые обязательные зависимости:**

- платные API;
- платные прокси;
- платный хостинг;
- платные AI API;
- пробные тарифы.

**Допускается:** необязательная интеграция, которую можно отключить без потери основной работоспособности.

## DEC-003. Архитектура

**Решение:** модульный монолит.

**Причина:** это проще для разработки, запуска, отладки и сопровождения, чем микросервисы.

**Пересмотр:** только после появления измеримой проблемы, которую монолит не решает.

## DEC-004. Backend

**Решение:** Python и FastAPI.

**Причина:** доступность библиотек, понятная структура и пригодность для локального веб-приложения.

## DEC-005. База данных

**Решение:** SQLite в первой версии.

**Причина:** не требует отдельной установки и подходит для одного локального пользователя.

**Обязательный будущий переход:** до появления многопользовательской веб-версии используется SQLite. PostgreSQL становится обязательной базой данных при переходе к многопользовательской веб-версии или раньше, если подтверждены ограничения SQLite. Сам переход выполняется отдельным проверяемым этапом, когда потребность возникнет, а не заранее.

## DEC-006. Интерфейс

**Решение:** серверный HTML и минимальный JavaScript в первой версии.

**Причина:** не усложнять проект отдельным frontend-приложением до появления необходимости.

**React не используется**, пока сложность интерфейса не докажет необходимость перехода.

## DEC-007. UI/UX

**Решение:** единый интерфейс с прогрессивным раскрытием функций.

**Основной принцип:**

- базовые функции видимы сразу;
- дополнительные встроены в существующие элементы;
- дополнительные возможности раскрываются при взаимодействии;
- они не должны визуально конкурировать с основным действием;
- отдельный «профессиональный режим» не создаётся без необходимости.

## DEC-008. Работа с AI

**Решение:** ChatGPT и Codex используются для разработки, но готовый продукт не зависит от платного AI API.

**Допускается позднее:** локальная AI-модель как необязательный модуль.

## DEC-009. Поисковая выдача

**Решение:** не использовать обход CAPTCHA, антибот-защит и платные SERP API как обязательную основу.

**Начальный подход:**

- ручной импорт;
- официальные бесплатные данные;
- ограниченный режим;
- явное отображение полноты и ограничений данных.

## DEC-010. Достоверность аналитики

**Решение:** система обязана разделять факт, метрику, корреляцию, гипотезу и рекомендацию.

**Запрещено:** называть гипотезу доказанной причиной без достаточных данных.

## DEC-011. Базовый документ требований

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать согласованный `REQUIREMENTS.md` версии 1.0 как подробное описание бизнес-, функциональных и нефункциональных требований.

**Последствия:**

- Codex обязан читать `REQUIREMENTS.md` до изменения кода;
- требования реализуются только в порядке `ROADMAP.md`;
- открытый вопрос согласуется до этапа, на который он влияет.

## DEC-012. ORM первой версии

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать SQLModel для работы с SQLite.

**Причина:** SQLModel соответствует утверждённому стеку и позволяет сохранить простую модель данных первой локальной версии.

**Возможный пересмотр:** переход на чистый SQLAlchemy допускается только по отдельному решению при наличии измеримой причины, например подтверждённого выигрыша в производительности, эффективности или сопровождаемости. Переход на PostgreSQL сам по себе не требует одновременной замены SQLModel.

## DEC-013. Первый вариант обходчика

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать безопасные значения по умолчанию: 200 страниц, глубина 3, задержка 1 секунда, таймаут 15 секунд и один одновременный запрос к домену.

Дополнительно:

- соблюдать запреты `robots.txt` без функции обхода запрета;
- использовать прозрачный User-Agent;
- не выполнять динамический JavaScript-рендеринг в первом варианте;
- добавлять динамический рендеринг только после подтверждённой необходимости.

## DEC-014. Сравнение текста и анализ структуры

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** очищать служебные блоки, сохранять букву `ё`, нормализовать Unicode и использовать нормализованное расстояние Левенштейна как базовую метрику изменения текста.

Для анализа структуры использовать графовый анализ и PageRank. На одном представлении графа показывать не более 100 узлов.

Семантический анализ допускается позднее только как необязательное бесплатное локальное улучшение.

## DEC-015. Импорт и источники поисковых данных

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** первыми поддержать согласованные CSV-отчёты Google Search Console и Яндекс Метрики с автоматическим определением разделителя, картой названий полей и обязательным экраном сопоставления столбцов.

Для поисковых данных использовать Google Search Console и Яндекс Вебмастер как безопасные источники данных о собственном сайте. Полный состав выдачи конкурентов доступен только через ручной импорт или отдельный подтверждённый безопасный источник.

## DEC-016. Планировщик и уведомления

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** в тестовой версии расписание работает только при открытом приложении. Почтовое уведомление отправляется через необязательный SMTP пользователя, а локальный журнал и интерфейс остаются обязательным резервным способом сообщения о сбое.

## DEC-017. Проверка данных и оценка гипотез

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать многофакторный скоринг и обязательную проверку готовности данных перед формированием гипотезы.

Если обязательного источника нет, соответствующая гипотеза не формируется, а пользователь получает список недостающих данных. Точные веса факторов утверждаются до этапа 10.

## DEC-018. Окончательное удаление карточки сайта

**Дата:** 2026-07-14

**Статус:** принято

**Решение:** на этапе 1 сайт удаляется окончательно из SQLite только после отдельного явного подтверждения и только изменяющим POST-запросом. Архивирование и восстановление не реализуются.

**Причина:** текущая карточка сайта ещё не имеет связанных обходов и истории; это позволяет выполнить FR-014 без преждевременной модели архива.

**Альтернативы:** мягкое удаление, архив с восстановлением или отложенное удаление.

**Последствия:**

- подтверждение обязано показывать название, URL и необратимость операции;
- отмена и GET-запрос не изменяют данные;
- удаляется только выбранная карточка сайта, без файлов, журналов и других сайтов;
- это решение не разрешает автоматическое каскадное удаление будущих обходов, снимков или истории.

**Условия пересмотра:** до добавления связанных обходов, снимков или истории необходимо отдельно определить правила сохранения, удаления и восстановления этих данных.

### DEC-019. Хранение и удаление истории проверок доступности

**Дата:** 2026-07-15

**Статус:** принято

**Решение:** каждую ручную проверку доступности с корректным токеном сохранять в отдельной таблице SQLite, связанной с сайтом. Хранить время начала и завершения, итоговый статус, пользовательское сообщение и полученные HTTP-коды `robots.txt` и стартовой страницы. При подтверждённом окончательном удалении сайта удалять сайт и всю его историю одной транзакцией.

**Причина:** история нужна после перезапуска для контроля результатов и ошибок, а оставлять записи без удалённого сайта нельзя. Явное предупреждение с количеством записей сохраняет требование подтверждённого удаления исторических данных.

**Альтернативы:** не сохранять результаты; хранить историю отдельными файлами; сохранять историю после удаления сайта; добавить отдельное удаление, архив или восстановление истории.

**Последствия:**

- экран проверки показывает историю от новой записи к старой и раскрывает технические подробности по запросу;
- подтверждение удаления сайта показывает число связанных записей и предупреждает об их необратимом удалении;
- отмена, GET, неверный токен и неизвестный сайт не изменяют сайт или историю;
- данные других сайтов сохраняются;
- содержимое страниц, снимки и параметры обхода в историю не входят;
- решение уточняет DEC-018 только для истории ручных проверок доступности.

**Условия пересмотра:** перед добавлением истории полного обхода, архивирования, восстановления или самостоятельного удаления истории принять отдельное решение о сроках хранения и связях данных.

### DEC-020. Хранение запусков обхода и удаление связанных данных сайта

**Дата:** 2026-07-15

**Статус:** принято

**Решение:** сохранять запуски обхода и упорядоченные метаданные обработанных страниц в SQLite до подтверждённого окончательного удаления сайта, без автоматического срока хранения. При таком удалении одной транзакцией удалять `AvailabilityCheck`, `CrawlPageRecord`, `CrawlRun` и `Site` выбранного сайта.

**Причина:** результаты обхода должны переживать перезапуск и оставаться целостно связанными с сайтом, а подтверждённое удаление не должно оставлять бесхозную историю. Отдельные точные количества проверок, запусков и записей страниц дают пользователю понятный объём необратимого действия.

**Альтернативы:** автоматический срок хранения; сохранение запусков после удаления сайта; отдельное удаление, архивирование или восстановление запусков; автоматическое каскадное удаление без явного предупреждения.

**Последствия:**

- сохраняются настройки запуска, итог, счётчики и упорядоченные метаданные страниц, но не HTML, текст страниц и найденные ссылки;
- отмена, GET, неверный токен и неизвестный сайт ничего не изменяют;
- данные другого сайта сохраняются;
- подтверждение удаления показывает отдельные количества проверок доступности, запусков обхода и записей страниц;
- отдельное удаление, архивирование и восстановление запусков не добавляются;
- решение уточняет DEC-018 и DEC-019 для данных обхода;
- будущие снимки этапа 3 этим решением не охватываются.

**Условия пересмотра:** перед добавлением автоматического срока хранения, самостоятельного управления запусками или снимков этапа 3 принять отдельное решение.

### DEC-021. Хранение снимков и переносимость слоя данных

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** использовать существующий `CrawlRun` как контейнер снимка сайта. Для каждой успешно разобранной HTML-страницы добавлять одну связанную с `CrawlPageRecord` запись `CrawlPageSnapshot` со временем проверки в UTC, `Title`, `Description`, `H1`, нормализованным текстом, SHA-256 и полным списком внутренних ссылок.

`CrawlPageSnapshot` создаётся отдельной таблицей со связью один к одному с `CrawlPageRecord`. Найденные внутренние ссылки сохраняются как канонический UTF-8 JSON-текст. Сырой HTML, изображения, цены и другие ещё не согласованные данные не сохраняются.

При завершённом запуске публикуются снимки всех успешно разобранных HTML-страниц. При частично завершённом запуске сохраняются снимки успешных страниц вместе с существующими записями ошибок, но такой запуск не становится базой будущего сравнения. Базой сравнения считается последний завершённый запуск. Неуспешный, отложенный или прерванный запуск не публикует новый набор снимков и не заменяет предыдущую базу.

Записи страниц и их снимки публикуются при финализации запуска одной транзакцией. Ранее сохранённые запуски автоматически не дополняются снимками.

Снимки хранятся без автоматического срока хранения до подтверждённого окончательного удаления сайта. При удалении сайта одной транзакцией удаляются снимки страниц, записи страниц, запуски обхода, проверки доступности и сам сайт. Подтверждение показывает отдельное количество страниц с сохранённым содержимым. Самостоятельное удаление, архивирование и восстановление снимков пока не добавляются.

Текущая реализация остаётся на SQLite и SQLModel. PostgreSQL обязателен к моменту перехода продукта в многопользовательскую веб-версию, но не внедряется до соответствующего этапа. Переход на чистый SQLAlchemy выполняется только по отдельному решению при наличии измеримой объективной причины и не объединяется автоматически с переходом на PostgreSQL.

До будущей миграции новые решения слоя данных обязаны:

- использовать переносимые типы данных, явные внешние ключи, ограничения уникальности и транзакционные границы;
- не применять SQLite-специфичные JSON-функции, типы или SQL в доменной логике;
- изолировать создание engine, session и особенности блокировок в слое хранения;
- не связывать crawler и разбор HTML напрямую с SQLModel или конкретным драйвером базы данных;
- предусматривать явные и обратимые изменения схемы без потери ранее собранных данных.

Будущий переход на PostgreSQL выполняется отдельной задачей с резервной копией и проверяемым переносом данных. Приёмка миграции должна подтвердить количество сайтов, запусков, страниц и снимков, целостность связей, UTC-время, SHA-256 и найденные ссылки.

**Причина:** отдельная добавочная таблица минимально изменяет действующую схему, позволяет старой версии приложения игнорировать новые данные и даёт этапу 4 стабильную основу сравнения по запуску, URL и SHA-256. Переносимые типы и изоляция слоя хранения снижают стоимость обязательного будущего перехода на PostgreSQL без преждевременного усложнения локальной версии.

**Альтернативы:** добавить поля снимка прямо в `CrawlPageRecord`; хранить ссылки в SQLite-специфичном JSON; создать отдельную нормализованную таблицу каждой ссылки; сохранять сырой HTML; внедрить PostgreSQL или чистый SQLAlchemy уже на этапе 3; не сохранять частичные результаты.

**Последствия:**

- TASK-0014 может добавить одну новую таблицу и атомарное сохранение согласованных данных без изменения crawler и парсера под конкретную СУБД;
- сохранённый частичный запуск остаётся доступной историей, но не подменяет надёжную базу сравнения;
- канонический JSON-текст достаточно прост для текущего лимита обхода и переносится между SQLite и PostgreSQL без обязательной смены типа;
- при будущей необходимости поиска и анализа отдельных ссылок формат хранения может быть изменён отдельной миграцией;
- решение уточняет DEC-005, DEC-012, DEC-018, DEC-019 и DEC-020, не запуская переход на PostgreSQL или SQLAlchemy сейчас.

**Условия пересмотра:** автоматический срок хранения, отдельное управление снимками, значительное увеличение лимитов обхода, необходимость запросов по отдельным ссылкам, многопользовательская веб-версия, подтверждённые ограничения SQLite или измеримая причина перехода с SQLModel на чистый SQLAlchemy требуют отдельного решения и миграционного плана.

### DEC-022. Целевое развёртывание на корпоративном сервере

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** готовый продукт до начала рабочего использования обязательно переносится на корпоративный сервер пользователя, использует его вычислительные ресурсы и поддерживает одновременную работу нескольких пользователей.

Текущие этапы остаются локальными и не получают преждевременно серверную инфраструктуру. Подготовка корпоративного развёртывания выделяется в обязательный завершающий этап после реализации и системной проверки основных функций. На этом этапе выполняются переход на PostgreSQL, многопользовательская аутентификация и авторизация, разделение данных, серверная координация длительных задач, эксплуатационная диагностика, резервное копирование и проверяемое развёртывание.

До серверного этапа каждое новое решение должно сохранять переносимость:

- конфигурация базы данных, путей, адреса, порта и секретов задаётся вне исходного кода;
- доменная логика, crawling, parsing, comparison и presentation не зависят от SQLite, конкретного драйвера, локального пути или одного типа окружения;
- особенности engine, session, транзакций, блокировок и файловой системы изолируются в инфраструктурном слое;
- новые длительные операции сохраняют достаточное состояние для восстановления и не полагаются на память процесса как на единственный источник истины;
- схема данных меняется явно и обратимо, а собранные данные можно перенести с проверкой количества записей и связей;
- не добавляются абсолютные Windows-пути и другие необязательные платформенные ограничения.

Это требование не разрешает заранее добавлять PostgreSQL, Redis, Celery, Kubernetes, микросервисы, контейнеризацию или другие инфраструктурные компоненты. Конкретный способ запуска на корпоративном сервере выбирается отдельным решением после получения параметров операционной системы, сети, безопасности и эксплуатации.

Переход на чистый SQLAlchemy не является обязательной частью серверного переноса и допускается только по отдельному решению при измеримой объективной пользе.

**Причина:** корпоративный сервер и одновременная работа нескольких пользователей являются обязательным целевым режимом продукта. Раннее внедрение неизвестной серверной инфраструктуры усложнило бы текущую локальную разработку, а игнорирование целевого режима создало бы дорогостоящую переделку в конце проекта.

**Альтернативы:** оставить продукт только локальным; считать сервер необязательным будущим направлением; внедрить весь серверный стек на текущем этапе; сразу перейти к микросервисам или обязательной публичной облачной платформе.

**Последствия:**

- ROADMAP получает обязательный завершающий этап корпоративного развёртывания;
- PostgreSQL остаётся обязательным для многопользовательской версии по DEC-005 и DEC-021;
- текущие задачи выполняются на SQLite и SQLModel, но не должны усиливать привязку бизнес-логики к ним;
- перед серверным этапом отдельно утверждаются параметры корпоративной среды, схема пользователей и ролей, способ исполнения фоновых задач, миграционный и эксплуатационный планы;
- готовность к рабочему использованию требует тестов конкурентности, перезапуска, миграции, backup/restore и документированного отката;
- решение уточняет DEC-001, DEC-003, DEC-005, DEC-012 и DEC-021.

**Условия пересмотра:** изменение целевой среды, требований безопасности, числа пользователей, способа корпоративной аутентификации или эксплуатационных ограничений требует отдельного решения, но не отменяет обязательную серверную готовность без явного решения пользователя.

### DEC-023. Надёжное извлечение цен из структурированных данных

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** на текущем этапе единственным дополнительно утверждённым элементом страницы по FR-049 является цена. Другие элементы добавляются только после отдельного явного согласования.

Первый слой обнаружения цен использует только структурированную разметку Schema.org в форматах JSON-LD и Microdata:

- `Offer.price` и связанная `priceCurrency`;
- `AggregateOffer.lowPrice` и `highPrice`;
- активная цена из `UnitPriceSpecification`, только если у того же предложения нет корректной прямой `Offer.price`.

Цена представляется отдельным неизменяемым значением с точной неотрицательной десятичной суммой, необязательной валютой, видом цены и источником. Сумма разбирается через `Decimal` без промежуточного `float`; допускается точка или однозначная запятая как десятичный разделитель, но не неоднозначные разделители тысяч, отрицательные и специальные числовые значения. Ноль является найденной ценой. Отсутствующая валюта остаётся отсутствующей; присутствующий корректный трёхбуквенный код приводится к верхнему регистру.

Обнаруженные значения упорядочиваются детерминированно. Точные дубликаты по виду, сумме и валюте удаляются с сохранением первого источника и порядка.

Отсутствующая, неподдерживаемая или повреждённая Schema.org-разметка не является ошибкой обхода или разбора страницы. Пустой список означает «цена надёжно не обнаружена», а не «на странице нет цены». Остальные данные страницы и её SHA-256 формируются как обычно; поэтому изменение видимой цены может изменить общий снимок текста, даже если цена не выделена отдельно.

На этом шаге не выполняется эвристический поиск чисел в видимом тексте: без контекста он может принять за цену доставку, рассрочку, старую цену, скидку, номер товара или другое число. Также не используются Open Graph, RDFa, исполнение JavaScript и специфичные для отдельных сайтов CSS-селекторы. Такие источники добавляются отдельным решением после проверки реальных целевых сайтов.

TASK-0015 ограничивается извлечением цен в `PageData` без изменения SQLite, SQLModel, интерфейса и сетевого слоя.

TASK-0016 сохраняет каждую обнаруженную цену отдельной `CrawlPagePriceRecord`, связанной «много цен к одному снимку страницы». Запись хранит порядок, вид, источник, необязательную валюту и канонический десятичный текст суммы. Кодирование не использует `float`, экспоненту, знак плюса и незначащие конечные нули: `10.00` становится `10`, `0.0100` — `0.01`, любой ноль — `0`. Пустой список цен представлен отсутствием связанных записей.

Записи страниц, снимки и цены публикуются при финализации запуска одной транзакцией. Завершённый и частичный запуск сохраняют цены успешных HTML-страниц; отложенный, аварийный и прерванный запуск не публикуют их. Подтверждение окончательного удаления сайта показывает отдельное количество сохранённых цен, а транзакционное удаление удаляет цены до снимков и остальных связанных данных.

Сумма пока хранится переносимым текстом, чтобы SQLite не преобразовывала точный `Decimal` через двоичное число. Будущий переход на PostgreSQL может сохранить этот формат либо выполнить отдельную проверяемую миграцию в `NUMERIC`, если появится необходимость в запросах и вычислениях на уровне базы данных.

**Причина:** структурированная разметка даёт явную семантику суммы, валюты и вида цены и позволяет получить воспроизводимый результат без ложных утверждений. Разделение извлечения и хранения сохраняет небольшие границы задач и независимость парсера от текущей базы данных.

**Альтернативы:** искать любые числа и валютные символы в видимом тексте; сразу поддержать все форматы разметки и JavaScript; считать отсутствие найденной цены доказательством отсутствия цены; объединить извлечение, хранение, интерфейс и сравнение в одной задаче.

**Последствия:**

- TASK-0015 добавляет детерминированный доменный результат цены без изменений схемы данных;
- страница без поддерживаемой разметки остаётся успешно обработанной с пустым списком цен;
- до TASK-0016 цены существуют только в результате обработки страницы и не сохраняются после завершения процесса;
- после TASK-0016 этап 3 может быть завершён, если дополнительные элементы страницы не будут отдельно утверждены;
- будущие резервные источники оцениваются на реальных примерах и не должны ухудшать достоверность уже поддерживаемых источников;
- решение сохраняет переносимость по DEC-021 и DEC-022 и не требует PostgreSQL или чистого SQLAlchemy.

**Условия пересмотра:** недостаточное покрытие Schema.org на реальных целевых сайтах, необходимость распознавать скидки, старые цены, наличие или несколько товарных сущностей, поддержка JavaScript-сайтов либо подтверждённые надёжные правила для Open Graph, RDFa или видимого HTML требуют отдельного решения и набора проверочных примеров.

### DEC-024. База сравнения и начальная классификация важности изменений

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** сравнивать только новый завершённый (`completed`) обход с непосредственно предыдущим завершённым обходом того же сайта. Частичные, неуспешные, отложенные и прерванные запуски сохраняются в истории, но не создают события и не становятся базой сравнения. Первый завершённый обход становится исходной базой без событий о добавлении всех его страниц.

Страницы сопоставляются по уже нормализованному URL. Новая или удалённая страница создаёт одно событие страницы без отдельных событий каждого её поля. Для страницы, присутствующей в обоих снимках, отдельно обнаруживаются изменения `Title`, `Description`, `H1`, нормализованного текста и множества внутренних ссылок. Отсутствующее значение отличается от пустой строки; изменение только порядка ссылок не считается изменением.

Каждое событие содержит тип, URL страницы, UTC-время завершения нового обхода, ссылки на старый и новый запуск и доступные ссылки на старую и новую версии страницы. Для новой или удалённой страницы отсутствующая сторона остаётся пустой. Повторная обработка одной пары запусков не должна создавать дубли событий.

Нормализованная доля изменения текста равна расстоянию Левенштейна, делённому на максимальную длину старого и нового нормализованного текста. Два пустых текста дают `0`, пустой и непустой — `1`. Для внутренних ссылок доля изменения равна размеру симметрической разности множеств, делённому на размер их объединения; два пустых множества не создают событие.

Числовые границы классов едины для текста и ссылок:

- `0` — события нет;
- больше `0`, но меньше `0,10` — низкая важность, вес `1`;
- от `0,10` включительно, но меньше `0,30` — средняя важность, вес `2`;
- от `0,30` включительно — высокая важность, вес `3`.

Тип поля задаёт минимальный вес события:

- новая страница — `2`;
- удалённая страница — `3`;
- изменение `Title` — `3`;
- изменение `H1` — `3`;
- изменение `Description` — `2`;
- изменение текста или внутренних ссылок — вес по числовому порогу.

Общая важность страницы равна максимальному весу её событий, а не сумме. Семантический анализ на базовом уровне не используется. Сравнение сохранённых цен не включается скрыто: для него отдельно согласуются сопоставление нескольких предложений и понятная трактовка состояния «цена надёжно не обнаружена».

**Причина:** надёжная база исключает ложные удаления из частичных обходов, отдельные события сохраняют объяснимость, а простые пороги и минимальные веса дают пользователю понятную первичную сортировку без обязательной AI-модели. Чистая доменная логика сравнения сохраняет переносимость к PostgreSQL и независимость от SQLModel.

**Альтернативы:** сравнивать с любым последним запуском; считать первый снимок набором новых страниц; суммировать веса событий; использовать только процент изменения без учёта поля; сразу добавить семантическую модель или сравнение цен.

**Последствия:**

- этап 4 реализуется небольшими задачами, начиная с чистого сопоставления страниц;
- выбор завершённых запусков и хранение событий остаются обязанностью слоя хранения, а не доменной функции;
- изменённая страница определяется наличием хотя бы одного отдельного события и не требует дублирующего общего события;
- реализация Левенштейна и её производительность проверяются отдельной задачей без изменения утверждённых порогов;
- цены остаются сохранёнными данными, но их отдельные события требуют следующего решения;
- решение уточняет DEC-021–DEC-023 и закрывает открытые вопросы раздела 12.2 `REQUIREMENTS.md`.

**Условия пересмотра:** реальные данные показывают слишком много шума или пропущенных важных изменений; частичные обходы необходимо использовать для ограниченных событий; требуется отдельная агрегированная оценка страницы; согласовано сравнение цен или семантический анализ.

### DEC-025. Обязательная комплексная пользовательская приёмка перед серверной миграцией

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** после реализации функций и системной проверки UX/UI, но до перехода на PostgreSQL и корпоративный сервер, продукт проходит отдельный обязательный этап комплексной пользовательской проверки и подготовки функционального релиз-кандидата.

Этап проверяет продукт целиком в условиях, максимально близких к реальной работе: все основные пользовательские сценарии и связанные требования, репрезентативные сайты и данные, целостность сохранённой информации, установку и обновление, перезапуск и восстановление после сбоев, длительные операции, согласованную реалистичную локальную нагрузку, поддерживаемые браузеры, доступность, понятность интерфейса и актуальность инструкций.

Найденные дефекты исправляются отдельными небольшими задачами. После исправлений выполняются целевые проверки и полная регрессия. Этап завершается только после явной пользовательской приёмки, отсутствия известных блокирующих и критических дефектов и документирования всех принятых неблокирующих ограничений.

На этом этапе не добавляется преждевременно серверная инфраструктура. Принятый релиз-кандидат остаётся локальным; миграция данных, PostgreSQL, аутентификация, роли, многопользовательская конкурентность и эксплуатационная проверка корпоративного сервера выполняются на следующем обязательном этапе и проходят отдельную приёмку.

**Причина:** разрозненные проверки небольших задач не заменяют достоверную сквозную проверку завершённого продукта на реалистичных сценариях. Отдельный приёмочный этап снижает риск переносить на корпоративную инфраструктуру функциональные дефекты, неполные сценарии или неудобный интерфейс и создаёт стабильную исходную точку для серверной миграции.

**Альтернативы:** ограничиться приёмкой отдельных задач; совместить функциональную приёмку с миграцией на сервер; начинать серверную подготовку до завершения функций; считать автоматические тесты достаточной заменой пользовательской проверке.

**Последствия:**

- в `ROADMAP.md` добавляется обязательный этап 12, а корпоративная подготовка становится этапом 13;
- текущие этапы сохраняют собственные проверки и приёмку, а новый этап дополняет их сквозной регрессией;
- конкретный план приёмки, тестовые данные и допустимая нагрузка согласуются перед началом этапа 12;
- переход к этапу 13 запрещён без явной приёмки функционального релиз-кандидата пользователем;
- DEC-022 уточняется последовательностью работ, но обязательность PostgreSQL и корпоративного многопользовательского развёртывания не изменяется;
- готовность к рабочему использованию наступает только после завершения и отдельной приёмки этапов 12 и 13.

**Условия пересмотра:** существенное изменение состава продукта, целевых пользовательских сценариев, требований к качеству, поддерживаемых окружений или корпоративной инфраструктуры требует обновить план приёмки, но не отменяет обязательный приёмочный шлюз без отдельного решения пользователя.

### DEC-026. Кроссплатформенная локальная версия

**Дата:** 2026-07-17

**Статус:** принято

**Решение:** локальная версия Marketing Intelligence должна предоставлять одинаковые функции на Windows 10/11, macOS и Linux при наличии Python 3.11 или новее и поддерживаемого настольного браузера.

Платформы могут иметь разные команды установки, активации виртуального окружения, настройки переменных окружения и запуска. Исходный код приложения, форматы данных и пользовательские возможности не разделяются на платформенные варианты. Пути, кодировка, файловые операции, процессы, сетевой слой и автоматические тесты должны оставаться переносимыми.

Подтверждённая совместимость указывается только после фактической проверки. На момент принятия решения запуск и пользовательский сценарий подтверждены только на Windows; инструкции для macOS и Linux добавляются с явным примечанием, что они ещё не проверены на реальных системах.

Одинаковая функциональность требуется для каждого нового этапа. Если изменение затрагивает платформенно-зависимую область, необходимо проверить доступные целевые окружения, а отсутствие фактической проверки зафиксировать как ограничение. Мобильный адаптивный интерфейс не включается в это решение и остаётся отдельным возможным направлением.

**Причина:** локальное веб-приложение состоит не только из браузерного интерфейса, но и из запускаемого на компьютере Python-сервера. Без явного требования Windows-инструкции могли бы незаметно превратиться в ограничение продукта, хотя утверждённый стек и архитектура допускают переносимость.

**Альтернативы:** поддерживать только Windows; отложить переносимость до завершения функций; создавать отдельные версии приложения для каждой операционной системы; считать браузерный интерфейс достаточной гарантией без проверки локального сервера.

**Последствия:**

- `PROJECT.md`, `REQUIREMENTS.md` и `AGENTS.md` закрепляют Windows 10/11, macOS и Linux как равные локальные целевые платформы;
- `README.md` содержит отдельные инструкции запуска, при этом непроверенные инструкции для macOS и Linux явно помечены;
- новые функции не могут зависеть от Windows-пути, оболочки или иной особенности одной платформы без равнозначного переносимого варианта;
- новые обязательные зависимости, отдельные кодовые версии и преждевременная серверная инфраструктура не добавляются;
- практическая проверка macOS и Linux выполняется при появлении доступных окружений и фиксируется отдельно; до этого документация не заявляет подтверждённую совместимость;
- после корпоративного развёртывания платформа компьютера пользователя не должна влиять на функциональность в поддерживаемом браузере, а параметры серверной ОС согласуются на корпоративном этапе (после DEC-027 — этап 14).

**Условия пересмотра:** изменение списка поддерживаемых настольных платформ, минимальной версии Python, браузерных требований, способа локальной установки или целевой корпоративной среды требует отдельного решения.

### DEC-027. Обязательный этап усиления безопасности приложения и защиты данных

**Дата:** 2026-07-20

**Статус:** принято

**Решение:** после системного улучшения UX/UI и до комплексной пользовательской приёмки продукт проходит отдельный обязательный этап усиления безопасности приложения и защиты данных.

На этом этапе все входы из браузера, импортов, интеграций, сетевых ответов, фоновых задач, команд и конфигурации рассматриваются как недоверенные. Проверка данных, разрешение доступа к объекту, изменение состояния, исходящие запросы и защита целостности выполняются сервером через явные прикладные границы и не зависят от frontend-проверок, скрытых полей или знания идентификатора.

Защита проектируется переносимо: прикладные правила не зависят от SQLite, локального пути, одного процесса или отсутствия аутентификации. Границы должны позволять позднее передать явный контекст пользователя, роли и организации без переписывания доменной логики. При этом сам этап не внедряет заранее PostgreSQL, корпоративную аутентификацию, Redis, reverse proxy, WAF и другие компоненты, параметры которых зависят от реальной среды.

Этап завершается только после согласованной модели угроз, проверки поверхности HTTP- и сетевых операций, потоков данных и зависимостей, автоматических негативных сценариев и фактической приёмки. Известный неустранённый критический или высокий риск по согласованной модели угроз блокирует переход к комплексной пользовательской приёмке.

После миграции на PostgreSQL и изменения архитектуры меры проверяются повторно в фактической корпоративной среде. На корпоративном этапе добавляются аутентификация, роли и разделение данных, TLS, сетевой периметр, минимальные привилегии PostgreSQL, корпоративное хранение секретов, защита рабочей базы и резервных копий по корпоративной политике, аудит доступа и необходимые распределённые ограничения.

**Причина:** отдельные безопасные проверки в небольших задачах не заменяют системную проверку всех границ доверия. Если выполнить её только после серверной миграции, ошибки прикладного слоя смешаются с рисками новой базы, многопользовательского доступа и инфраструктуры; если выполнить только до миграции, изменение архитектуры может создать новые пути обхода. Два последовательных барьера уменьшают оба риска.

**Альтернативы:** ограничиться текущими проверками каждой функции; совместить безопасность с общей пользовательской приёмкой; отложить всю защиту до корпоративного сервера; заранее внедрить полную серверную инфраструктуру.

**Последствия:**

- в `ROADMAP.md` добавляется новый этап 12; прежняя комплексная пользовательская приёмка становится этапом 13, а корпоративная подготовка — этапом 14;
- текущие и будущие функции продолжают получать обычные проверки безопасности, а этап 12 выполняет системный аудит и устранение межмодульных рисков;
- конкретные задачи этапа формируются по модели угроз и выполняются небольшими проверяемыми изменениями;
- план проверки использует применимые требования актуальной стабильной OWASP ASVS и практики NIST SSDF с зафиксированной версией и явным обоснованием исключений;
- пользовательская приёмка этапа 13 включает регрессию усиленной защиты;
- корпоративный этап 14 повторяет проверку после PostgreSQL, аутентификации, многопроцессного запуска и изменения сетевой архитектуры;
- абсолютная неуязвимость не заявляется: критерий перехода — отсутствие известных неустранённых критических и высоких рисков в согласованной области проверки.

DEC-025 сохраняет обязательность пользовательского приёмочного шлюза, но его актуальные номера этапов после этого решения — 13 для релиз-кандидата и 14 для корпоративной подготовки.

**Условия пересмотра:** изменение модели угроз, типов обрабатываемых данных, способов импорта и интеграций, режима доступа, корпоративной аутентификации, сетевой среды или требований информационной безопасности требует обновить меры и план проверки, но не отменяет оба барьера без отдельного решения пользователя.

### DEC-028. Достоверное сравнение цены по товарной посадочной странице

**Дата:** 2026-07-20

**Статус:** принято

**Решение:** на первом уровне истории цен товар идентифицируется нормализованным URL посадочной страницы. `Title` может показываться как пояснение, но не участвует в сопоставлении. Цена сравнивается только для страницы, присутствующей в двух последовательных completed-обходах одного сайта и имеющей один из двух однозначных профилей на обеих сторонах:

- ровно одна обычная цена `price` с непустой валютой;
- ровно один диапазон из `low` и `high` с одной непустой валютой, неотрицательными значениями и `low <= high`.

Профиль и валюта должны совпадать. Суммы сравниваются как точные `Decimal` без округления. Порядок записей и источник JSON-LD или Microdata не участвуют в идентичности. Одинаковые суммы при другом источнике или порядке не создают событие; изменение одной или обеих сравнимых сумм создаёт ровно одно событие цены для страницы.

Система не утверждает, что цена изменилась, если страница новая или удалена, цена или валюта отсутствует, профиль или валюта различаются, обычных цен несколько, присутствуют лишние или неоднозначные предложения, значение отрицательное или повреждено. Эти состояния означают только отсутствие надёжного сравнения, а не отсутствие изменения в реальном мире.

Ценовые события хранятся в отдельной добавочной таблице со ссылками на оба запуска и обе версии страницы. Это сохраняет действующее ограничение семи типов `SnapshotChangeEvent` и позволяет открыть существующую SQLite без перестройки таблицы. Общая и отдельная ленты объединяют обычные и ценовые события через переносимый query-слой с порядком `current_completed_at DESC, source_rank ASC, event_id DESC`, где обычные события имеют меньший `source_rank`, и общей пагинацией. Обычная страница выполняет один общий `count` и один объединённый `select` без N+1. Точные значения загружаются из связанных снимков, а чтение не изменяет базу.

Событие получает тип «Изменение цены», точное время нового completed-обхода, валюту и посадочную страницу. Для обычной цены показывается одно значение, для диапазона — нижняя и верхняя границы. Важность не оценивается и явно показывается как «Не оценивалась». Подробности соблюдают утверждённый порядок `Стало` над `Было` и безопасно сохраняют состояние возврата.

**Причина:** сохранённые данные пока содержат сумму, валюту, вид, источник и страницу, но не надёжный SKU или идентификатор каждого товара. Консервативный профиль URL позволяет показать полезные достоверные изменения типичных товарных страниц и диапазонов без ложного сопоставления нескольких товаров, старой и акционной цены или разных валют.

**Альтернативы:** сопоставлять любые списки цен по позиции; считать `Title` идентификатором товара; сравнивать разные валюты; трактовать исчезновение цены как изменение; расширить существующую таблицу событий миграцией; отложить всю историю цен до извлечения SKU.

**Последствия:**

- TASK-0030 реализует одну цельную вертикаль от чистого сравнения до общей и отдельной лент и подробностей;
- первый completed-обход остаётся базовой линией, остальные статусы не создают событие и не меняют базу сравнения;
- повторная обработка пары должна быть идемпотентной;
- окончательное удаление сайта удаляет связанные ценовые события той же транзакцией по DEC-020 и не затрагивает другие сайты;
- существующие старые пары обходов ретроспективно не обрабатываются; новая таблица начинает наполняться при обработке последующих completed-обходов;
- сложные страницы с несколькими товарами, обычными ценами или ролями вроде старой и акционной цены сознательно пропускаются;
- новые зависимости, JavaScript, экспорт и отметка просмотра не добавляются.

**Условия пересмотра:** появление надёжно сохранённых SKU, product ID, Offer ID, роли цены или связи цены с конкретной сущностью товара; потребность в ретроспективной обработке; подтверждённая необходимость сравнивать сложные предложения, валютные изменения или оценивать важность цены требуют отдельного решения.

### DEC-029. Локальное состояние просмотра и безопасный экспорт истории

**Дата:** 2026-07-20

**Статус:** принято

**Решение:** событие считается непросмотренным, пока для него нет отдельной записи состояния. Пользователь может только явным серверным POST-действием отметить обычное или ценовое событие просмотренным и вернуть его в непросмотренные. Открытие списка, подробностей или экспорта статус не меняет. Повторная установка того же состояния идемпотентна, время `viewed_at` хранится в UTC и обновляется только при фактическом переходе из непросмотренного состояния.

Состояние локальной версии хранится в новой добавочной таблице `ChangeEventViewState`. Одна строка содержит `viewed_at` и ровно одну из двух внешних ссылок: на `SnapshotChangeEvent` либо на `PriceChangeEvent`; для каждой ссылки действует уникальность и проверка взаимоисключения. Существующие таблицы событий и их ограничения не меняются, поэтому старая SQLite получает только отсутствующую таблицу, а вся прежняя история по умолчанию остаётся непросмотренной. При окончательном удалении сайта состояния удаляются до связанных событий той же транзакцией по DEC-020.

Прикладной сервис состояния принимает проверенные сайт, источник и идентификатор события и изолирован от HTTP, SQLModel и особенностей SQLite настолько, насколько это допускает слой хранения. POST-токен связывается с сайтом, источником, событием и требуемым переходом; необработанный URL возврата от клиента не принимается. В корпоративной многопользовательской версии это хранилище заменяется или расширяется ключом аутентифицированного пользователя и арендатора, а прикладная граница получает контекст доступа. До этапа 14 учётные записи и фиктивные идентификаторы пользователей не добавляются.

Объединённый query-слой включает `viewed_at`, состояние и фильтр «Все / Непросмотренные / Просмотренные» для обоих источников, сохраняя утверждённый порядок и два запроса обычной страницы. Статус показывается явным текстом в лентах и подробностях; изменение состояния сохраняет только проверенные фильтры, страницу и область возврата.

JSON и CSV экспортируют всю совпавшую с текущими фильтрами историю независимо от пагинации. Оба формата сохраняют порядок `current_completed_at DESC, source_rank ASC, event_id DESC`, включают однозначную пару `source` + `event_id`, сайт, тип, время, URL, важность, объяснение, состояние просмотра и точные стороны `Стало` и `Было`. Подготовка выполняется ограниченными пакетами и не допускает отдельного запроса на каждое событие или полной неограниченной материализации в памяти.

JSON использует стабильную машинную схему `schema_version: 1`, английские ключи, UTC ISO 8601 и структурированные состояния, сохраняющие различие отсутствующей стороны, `None`, пустой строки, пустого списка ссылок, текста, ссылок, одной цены и диапазона. CSV предназначен для бизнес-пользователя: русские заголовки, настроенное локальное время с явным смещением, человекочитаемые точные значения, стандартное цитирование, CRLF и обязательный UTF-8 BOM. Текстовые ячейки, начинающиеся с `=`, `+`, `-`, `@`, табуляции или возврата каретки, нейтрализуются перед открытием в электронных таблицах; исходные значения в JSON и базе не изменяются. Пустая выборка остаётся корректным файлом, а повреждённые данные не должны приводить к частичному файлу, утечке внутренних сведений или записи в базу.

**Причина:** отдельное состояние сохраняет совместимость старой базы и не смешивает пользовательское действие с неизменяемым фактом обнаруженного события. Явная, а не автоматическая отметка соответствует FR-064 и не скрывает непрочитанные изменения. Общий экспортный слой выполняет FR-065 сразу для обоих источников, повторно использует утверждённые фильтры и точные снимки, а пакетная обработка и нейтрализация формул делают результат пригодным для будущей серверной архитектуры и безопасного практического использования.

**Альтернативы:** добавить `viewed` в обе таблицы событий; автоматически отмечать событие при открытии; хранить фиктивного локального пользователя; использовать одну полиморфную ссылку без внешних ключей; экспортировать только текущую страницу или только сводные поля; собирать весь файл в памяти; добавлять XLSX, ZIP или стороннюю библиотеку.

**Последствия:**

- TASK-0031 одной вертикалью реализует состояние, фильтр, защищённые действия, общую и отдельную ленты, JSON и CSV и завершает этап 5 после приёмки;
- новая таблица должна безопасно создаваться на существующей SQLite и удаляться вместе с сайтом;
- экспорт и все GET-операции остаются read-only, а изменение статуса является отдельной защищённой транзакцией;
- формат JSON версии 1 становится публичным локальным контрактом автоматизации и меняется несовместимо только с новой версией схемы;
- CSV защищает Excel от интерпретации недоверенных значений как формул, но JSON сохраняет исходный текст без такой презентационной модификации;
- автоматическая и массовая отметка, импорт, XLSX/ZIP, плановый экспорт, учётные записи, роли, PostgreSQL, новые зависимости и JavaScript не добавляются.

**Условия пересмотра:** появление аутентифицированных пользователей и арендаторов, требования корпоративного разграничения доступа, необходимость массовой обработки, иной формат резервной копии или обратного импорта, подтверждённая потребность в XLSX либо измеренная проблема производительности экспорта требуют отдельного решения.

### DEC-030. Правила PageRank и структурных аномалий

**Дата:** 2026-07-20

**Статус:** принято

**Решение:** анализировать полный граф HTML-страниц одного выбранного `completed` или `partial`-обхода независимо от текущих фильтров интерфейса. В граф включаются только уникальные внутренние рёбра между сохранёнными HTML-страницами; остальные фактические состояния продолжают показываться, но не участвуют в PageRank и структурных сигналах. Частичный или ограниченный обход сопровождается предупреждением о возможной неполноте результата.

PageRank рассчитывается в переносимом доменном слое без новой зависимости: коэффициент затухания 0,85, начальный вес `1 / N`, равномерное распределение по исходящим рёбрам и равномерное перераспределение веса страниц без исходящих HTML-рёбер. Расчёт завершается при сумме абсолютных изменений не более `1e-12`, но не позднее 200 итераций; несходимость даёт контролируемую ошибку без ложного значения. Место при равных значениях определяется стабильным порядком обхода.

Недостаточная внутренняя связность применяется только минимум к 10 достижимым кандидатам без стартовой страницы и структурных островов. Сигнал получает страница с PageRank строго ниже P20 по методу ближайшего ранга и не более чем одной уникальной входящей HTML-ссылкой. Равенство порогу сигнал не создаёт.

Структурные сигналы определяются так:

- тупиковая зона — компонента HTML-графа без исходящего ребра в другую компоненту; одиночная страница без исходящих HTML-рёбер показывается как тупиковая страница;
- структурный остров — HTML-страница, недостижимая от стартовой страницы;
- циклическая ловушка — тупиковая сильно связная компонента минимум из двух страниц либо с самоссылкой;
- аномальная глубина применяется минимум к 10 достижимым HTML-страницам и требует глубину не меньше 5 и строго выше P90 по методу ближайшего ранга;
- бутылочное горлышко — не стартовая достижимая страница, исключение которой делает недостижимыми не менее `max(2, ceil(10% × (R - 1)))` других ранее достижимых страниц.

PageRank и сигналы вычисляются при чтении, не сохраняются в БД и показываются как расчётные показатели текущей карты, а не доказанная важность страницы, причина или рекомендация.

**Причина:** относительные и строгие пороги остаются полезными для сайтов разного размера, снижают ложные массовые сигналы на одинаковых значениях и допускают понятное воспроизводимое объяснение без зависимости от конкретной СУБД. Минимальная аномальная глубина 5 не помечает обычную цепочку «главная — каталог — подкаталог — карточка товара» глубиной 4.

**Альтернативы:** фиксированный абсолютный PageRank; включение non-HTML результатов; нестрогие P20 и P90; аномальная глубина от 3 или 4; сохранение рассчитанных значений в отдельной таблице; сторонняя графовая библиотека; автоматические рекомендации по перелинковке.

**Последствия:** TASK-0033 завершает этап 6 одним read-only аналитическим слоем, сводкой, серверным фильтром и подробностями. Схема БД, зависимости, crawler, два запроса query-слоя и лимит графа 100 узлов не изменяются. Сравнение структуры между обходами, история показателей, экспорт и автоматические рекомендации остаются за границами решения.

**Условия пересмотра:** значительное увеличение допустимого числа страниц, измеренная проблема производительности, другой способ формирования карты, появление нескольких стартовых страниц, необходимость хранить историю показателей или подтверждённые ложные сигналы на реальных сайтах требуют повторной проверки алгоритмов и порогов.

### DEC-031. Локальный планировщик, сохраняемая очередь и SMTP-уведомления

**Дата:** 2026-07-21

**Статус:** принято

**Решение:** для каждого сайта хранить не более одного выключенного по умолчанию расписания существующего полного обхода. Поддерживаются ежедневное и еженедельное правила; безопасное начальное значение — еженедельно в 09:00 текущего локального дня. Локальный ввод переводится в timezone-aware UTC, а первый запуск всегда назначается строго на будущее. Изменение или отключение правила отменяет его ещё не начавшиеся записи, но не прерывает уже выполняющийся обход.

APScheduler стабильной ветки 3.x используется только как внутрипроцессный таймер, который раз в секунду будит прикладной диспетчер FastAPI. Расписание, следующий момент, очередь и журнал хранятся в проектной БД и не дублируются в job store APScheduler. Планировщик работает без открытого браузера, пока запущен процесс FastAPI. При полностью остановленном процессе проверки не выполняются; после запуска каждый просроченный график получает одну сводную запись `missed` с числом пропущенных периодов и новым будущим моментом без catch-up и письма.

Наступившие расписания и ручные повторы образуют одну сохраняемую очередь. В локальном приложении одновременно выполняется не более одного полного обхода, стабильный порядок определяется плановым временем, ID сайта и ID записи. Занятый обычным ручным обходом crawler оставляет запись в ожидании. Уникальность пары «расписание — плановый момент» и транзакционная локальная защита исключают повторную постановку. Записи `pending` и `running`, оставшиеся после остановки процесса, при следующем запуске становятся `interrupted` и автоматически не продолжаются.

Добавляются только две таблицы: одно правило сайта и журнал, одновременно являющийся очередью. Журнал хранит снимок настроек crawler, время, статус, сообщение, необязательные ссылки на расписание, `CrawlRun` и исходный ручной повтор, число пропусков и отдельное состояние уведомления. Пользовательские состояния: `pending`, `running`, `completed`, `partial`, `deferred`, `failed`, `interrupted`, `missed`, `cancelled`. Ручной повтор разрешён только для `partial`, `deferred`, `failed` и `interrupted`, использует исходный снимок настроек и не сдвигает расписание.

SMTP полностью необязателен и читается только из `MI_SMTP_HOST`, `MI_SMTP_SECURITY`, `MI_SMTP_PORT`, `MI_SMTP_USERNAME`, `MI_SMTP_PASSWORD`, `MI_SMTP_FROM`, `MI_SMTP_TO` и `MI_SMTP_TIMEOUT`. Разрешены только STARTTLS и SSL, один отправитель и один получатель; логин и пароль задаются вместе. Пароль не сохраняется в БД, HTML или журнале. Стандартные `smtplib`, `ssl` и `email` вызываются вне основного async-потока. Одно текстовое письмо отправляется только для `partial`, `deferred`, `failed` и `interrupted`; ошибка доставки фиксируется отдельно и не меняет итог обхода. Реальный почтовый провайдер не считается проверенным без пользовательских реквизитов.

**Причина:** один внутрипроцессный диспетчер и две переносимые таблицы полностью решают локальную задачу без отдельной службы, worker, брокера сообщений или второго хранилища расписаний. Использование существующих `start_crawl_run` и `execute_crawl_run` сохраняет единую обработку снимков, сравнений и событий, а явные ограничения не создают ложного ожидания автономной серверной работы.

**Альтернативы:** системный cron или Windows Task Scheduler; APScheduler job store как источник истины; несколько расписаний одного сайта; catch-up всех пропущенных запусков; параллельные обходы; Redis, Celery или отдельный worker; обязательный почтовый сервис; HTML-письма, несколько получателей и OAuth; немедленный запуск при включении расписания.

**Последствия:** TASK-0034 завершает этап 7. Добавляется бесплатная зависимость `APScheduler>=3.11.3,<4.0` с лицензией MIT, две таблицы и интерфейс расписания. Существующая SQLite открывается без потери данных. Главная страница пакетно показывает состояние автоматизации без N+1, а все GET-просмотры остаются read-only. Работа при остановленном FastAPI, межпроцессная координация, PostgreSQL, аутентификация, роли и окончательная серверная эксплуатация не добавляются и остаются за этапом 14.

**Условия пересмотра:** переход к нескольким веб-процессам или корпоративному серверу, миграция на PostgreSQL, необходимость высокой доступности, несколько правил одного сайта, другие частоты, несколько получателей, OAuth или подтверждённая проблема производительности требуют отдельного решения и замены локальной координации без изменения доменной модели полного обхода.

### DEC-032. Безопасный импорт Google Search Console `Pages.csv`

**Дата:** 2026-07-22

**Статус:** принято

**Решение:** первым законченным импортом этапа 8 поддержать только CSV отчёта Google Search Console «Эффективность → Страницы» без дополнительных фильтров. Пользователь выбирает сайт, файл, календарный период и явно подтверждает тип отчёта. Обязательные логические поля — страница, клики и показы; средняя позиция необязательна. CTR не импортируется и не сохраняется: он рассчитывается из точных целых кликов и показов. Позиция хранится канонической десятичной строкой без использования `float`.

CSV разбирается стандартным модулем Python. Поддерживаются UTF-8 с BOM и без него, запятая, точка с запятой и табуляция; разделитель определяется через `csv.Sniffer`. Сервер ограничивает потоковое чтение 5 МиБ, 10 000 строк данных, 50 столбцов и длину недоверенных значений. Русские и английские синонимы заголовков дают только предложения; перед записью пользователь видит экранированный предпросмотр и может исправить сопоставление. Исходный файл не сохраняется.

Неподтверждённый предпросмотр хранится не более 30 минут в ограниченном потокобезопасном состоянии конкретного экземпляра приложения, привязан к сайту и защищён случайным токеном. Подтверждение выполняется одноразовым защищённым POST. Весь набор проверяется до транзакции; URL нормализуются, обязаны принадлежать origin выбранного сайта и не вызывают сетевых запросов. Любая ошибка отменяет создание журнала и изменение показателей.

Добавляются две переносимые таблицы: журнал подтверждённых импортов и текущие показатели страниц по периоду. Уникальность показателя — сайт, начало периода, конец периода и нормализованный URL. Повторный импорт создаёт новую запись журнала, добавляет новые строки, обновляет изменившиеся, считает совпавшие и не удаляет отсутствующие строки автоматически. Состояние относительно последнего `completed` или `partial`-обхода рассчитывается при чтении пакетно и не называется доказательством сиротской страницы.

FastAPI multipart-загрузка использует единственную новую зависимость `python-multipart>=0.0.32,<1.0` с лицензией Apache-2.0. Импорт индексирования Search Console, Яндекс Метрики, XLS/XLSX, API, фоновые задания, графики и аналитические гипотезы остаются за следующими задачами этапа 8 и более поздними этапами.

**Причина:** один ограниченный вертикальный сценарий даёт пользователю полезные фактические показатели страниц без платного API, одновременно проверяет общую архитектуру будущих CSV-импортов и не смешивает форматы с разными полями и правилами достоверности.

**Альтернативы:** импортировать сразу все отчёты этапа 8; сохранять исходный файл; доверять заголовкам без подтверждения; принимать CTR из CSV; удалять отсутствующие строки; использовать Pandas или другую CSV-библиотеку; подключить Search Console API или OAuth; хранить предпросмотр в БД.

**Последствия:** TASK-0035 открывает, но не завершает этап 8. Существующая SQLite получает две новые таблицы без потери данных; GET-просмотры остаются read-only, удаление сайта удаляет журнал и показатели в прежней транзакции. Временное состояние предпросмотра сознательно не переживает перезапуск и не предназначено для нескольких веб-процессов; корпоративный этап потребует заменить его общим хранилищем или иным подтверждаемым механизмом.

**Условия пересмотра:** новый формат экспорта Google, увеличение объёма файлов, импорт других отчётов, переход к нескольким веб-процессам, PostgreSQL, аутентификации, арендаторам, фоновой обработке или API требует отдельного решения и повторной проверки лимитов, идемпотентности и разграничения доступа.

### DEC-033. Разделение собственных сайтов и конкурентов

**Дата:** 2026-07-22

**Статус:** принято

**Решение:** внутри одного модульного монолита использовать два раздельных пользовательских инструмента: «Конкуренты» и «Свои сайты». Каждый сайт имеет обязательный строковый тип `competitor` или `owned` с именованным переносимым ограничением. Списки, глобальные истории, фильтры и экспорты всегда ограничиваются текущим типом, а обход, снимки, сравнение, карта структуры, расписание, редактирование и удаление используют общее ядро.

Существующая SQLite получает поле типа без пересоздания таблицы: сайты с подтверждёнными импортами или показателями Search Console становятся `owned`, остальные — `competitor`. Идентификаторы и все связанные данные сохраняются, повторная подготовка базы идемпотентна. SQLite-специфичная операция изолирована в подготовке локальной базы.

Импорт и показатели Search Console доступны только собственным сайтам. Тип повторно проверяется на каждом GET и POST, включая подтверждение ранее созданного предпросмотра. Пользователь может защищённым подтверждением перенести сайт между инструментами без изменения ID; перенос `owned` в `competitor` запрещён при наличии импортов или показателей Search Console и не удаляет данные.

Следующими отдельными задачами этапа 8 должны быть необязательные интеграции Search Console и Яндекс Метрики с периодическим обновлением. Ручной импорт сохраняется как резервный сценарий. В локальной версии автоматическое обновление возможно только при работающем приложении; постоянная серверная работа относится к корпоративному этапу.

**Причина:** закрытые показатели Search Console и Метрики относятся к собственным сайтам и не могут быть получены публичным обходом конкурента. Разделение предотвращает смешивание разных пользовательских задач и данных, сохраняя простую архитектуру и уже реализованное техническое ядро.

**Альтернативы:** оставить единый список и скрывать отдельные кнопки; создать второе приложение или базу; дублировать обход и историю; удалить GSC-данные при переносе; сразу добавить OAuth и внешние API. Единый список создаёт продуктовую путаницу, а остальные варианты преждевременно усложняют архитектуру или создают риск потери данных.

**Последствия:** TASK-0036 меняет навигацию и безопасно классифицирует существующие сайты. Старые GET-адреса импорта остаются только перенаправлениями для собственных сайтов, а прежняя общая история ведёт к истории конкурентов. OAuth, токены, API, Яндекс Метрика и фоновые синхронизации не добавляются. Будущая аутентификация и арендаторы должны дополнить фильтр типа проверкой владельца, не заменяя её.

**Условия пересмотра:** появление новых классов наблюдаемых ресурсов, подтверждённая необходимость совместного представления данных, многопользовательская модель, PostgreSQL или изменение правил хранения внешних данных требуют отдельного решения и повторной проверки миграции и разграничения доступа.

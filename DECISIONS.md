# DECISIONS.md — Журнал решений

Этот файл фиксирует принятые продуктовые и технические решения.

## DEC-001. Формат продукта

**Решение:** первая версия является локальным веб-приложением.

**Причина:** продукт должен работать без обязательного платного хостинга и открываться через обычный браузер.

**Последствия:**

- пользователь запускает приложение на своём компьютере;
- доступ через интернет не гарантируется;
- перенос на сервер рассматривается позднее.

## DEC-002. Финансовая модель разработки

**Решение:** основные функции должны работать без платных ресурсов.

**Запрещённые обязательные зависимости:**

- платные API;
- платные прокси;
- платный хостинг;
- платные AI API;
- пробные тарифы.

**Допускается:** необязательная интеграция, которую можно отключить без потери основной работоспособности.

## DEC-003. Архитектура

**Решение:** модульный монолит.

**Причина:** это проще для разработки, запуска, отладки и сопровождения, чем микросервисы.

**Пересмотр:** только после появления измеримой проблемы, которую монолит не решает.

## DEC-004. Backend

**Решение:** Python и FastAPI.

**Причина:** доступность библиотек, понятная структура и пригодность для локального веб-приложения.

## DEC-005. База данных

**Решение:** SQLite в первой версии.

**Причина:** не требует отдельной установки и подходит для одного локального пользователя.

**Обязательный будущий переход:** до появления многопользовательской веб-версии используется SQLite. PostgreSQL становится обязательной базой данных при переходе к многопользовательской веб-версии или раньше, если подтверждены ограничения SQLite. Сам переход выполняется отдельным проверяемым этапом, когда потребность возникнет, а не заранее.

## DEC-006. Интерфейс

**Решение:** серверный HTML и минимальный JavaScript в первой версии.

**Причина:** не усложнять проект отдельным frontend-приложением до появления необходимости.

**React не используется**, пока сложность интерфейса не докажет необходимость перехода.

## DEC-007. UI/UX

**Решение:** единый интерфейс с прогрессивным раскрытием функций.

**Основной принцип:**

- базовые функции видимы сразу;
- дополнительные встроены в существующие элементы;
- дополнительные возможности раскрываются при взаимодействии;
- они не должны визуально конкурировать с основным действием;
- отдельный «профессиональный режим» не создаётся без необходимости.

## DEC-008. Работа с AI

**Решение:** ChatGPT и Codex используются для разработки, но готовый продукт не зависит от платного AI API.

**Допускается позднее:** локальная AI-модель как необязательный модуль.

## DEC-009. Поисковая выдача

**Решение:** не использовать обход CAPTCHA, антибот-защит и платные SERP API как обязательную основу.

**Начальный подход:**

- ручной импорт;
- официальные бесплатные данные;
- ограниченный режим;
- явное отображение полноты и ограничений данных.

## DEC-010. Достоверность аналитики

**Решение:** система обязана разделять факт, метрику, корреляцию, гипотезу и рекомендацию.

**Запрещено:** называть гипотезу доказанной причиной без достаточных данных.

## DEC-011. Базовый документ требований

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать согласованный `REQUIREMENTS.md` версии 1.0 как подробное описание бизнес-, функциональных и нефункциональных требований.

**Последствия:**

- Codex обязан читать `REQUIREMENTS.md` до изменения кода;
- требования реализуются только в порядке `ROADMAP.md`;
- открытый вопрос согласуется до этапа, на который он влияет.

## DEC-012. ORM первой версии

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать SQLModel для работы с SQLite.

**Причина:** SQLModel соответствует утверждённому стеку и позволяет сохранить простую модель данных первой локальной версии.

**Возможный пересмотр:** переход на чистый SQLAlchemy допускается только по отдельному решению при наличии измеримой причины, например подтверждённого выигрыша в производительности, эффективности или сопровождаемости. Переход на PostgreSQL сам по себе не требует одновременной замены SQLModel.

## DEC-013. Первый вариант обходчика

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать безопасные значения по умолчанию: 200 страниц, глубина 3, задержка 1 секунда, таймаут 15 секунд и один одновременный запрос к домену.

Дополнительно:

- соблюдать запреты `robots.txt` без функции обхода запрета;
- использовать прозрачный User-Agent;
- не выполнять динамический JavaScript-рендеринг в первом варианте;
- добавлять динамический рендеринг только после подтверждённой необходимости.

## DEC-014. Сравнение текста и анализ структуры

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** очищать служебные блоки, сохранять букву `ё`, нормализовать Unicode и использовать нормализованное расстояние Левенштейна как базовую метрику изменения текста.

Для анализа структуры использовать графовый анализ и PageRank. На одном представлении графа показывать не более 100 узлов.

Семантический анализ допускается позднее только как необязательное бесплатное локальное улучшение.

## DEC-015. Импорт и источники поисковых данных

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** первыми поддержать согласованные CSV-отчёты Google Search Console и Яндекс Метрики с автоматическим определением разделителя, картой названий полей и обязательным экраном сопоставления столбцов.

Для поисковых данных использовать Google Search Console и Яндекс Вебмастер как безопасные источники данных о собственном сайте. Полный состав выдачи конкурентов доступен только через ручной импорт или отдельный подтверждённый безопасный источник.

## DEC-016. Планировщик и уведомления

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** в тестовой версии расписание работает только при открытом приложении. Почтовое уведомление отправляется через необязательный SMTP пользователя, а локальный журнал и интерфейс остаются обязательным резервным способом сообщения о сбое.

## DEC-017. Проверка данных и оценка гипотез

**Дата:** 2026-07-13

**Статус:** принято

**Решение:** использовать многофакторный скоринг и обязательную проверку готовности данных перед формированием гипотезы.

Если обязательного источника нет, соответствующая гипотеза не формируется, а пользователь получает список недостающих данных. Точные веса факторов утверждаются до этапа 10.

## DEC-018. Окончательное удаление карточки сайта

**Дата:** 2026-07-14

**Статус:** принято

**Решение:** на этапе 1 сайт удаляется окончательно из SQLite только после отдельного явного подтверждения и только изменяющим POST-запросом. Архивирование и восстановление не реализуются.

**Причина:** текущая карточка сайта ещё не имеет связанных обходов и истории; это позволяет выполнить FR-014 без преждевременной модели архива.

**Альтернативы:** мягкое удаление, архив с восстановлением или отложенное удаление.

**Последствия:**

- подтверждение обязано показывать название, URL и необратимость операции;
- отмена и GET-запрос не изменяют данные;
- удаляется только выбранная карточка сайта, без файлов, журналов и других сайтов;
- это решение не разрешает автоматическое каскадное удаление будущих обходов, снимков или истории.

**Условия пересмотра:** до добавления связанных обходов, снимков или истории необходимо отдельно определить правила сохранения, удаления и восстановления этих данных.

### DEC-019. Хранение и удаление истории проверок доступности

**Дата:** 2026-07-15

**Статус:** принято

**Решение:** каждую ручную проверку доступности с корректным токеном сохранять в отдельной таблице SQLite, связанной с сайтом. Хранить время начала и завершения, итоговый статус, пользовательское сообщение и полученные HTTP-коды `robots.txt` и стартовой страницы. При подтверждённом окончательном удалении сайта удалять сайт и всю его историю одной транзакцией.

**Причина:** история нужна после перезапуска для контроля результатов и ошибок, а оставлять записи без удалённого сайта нельзя. Явное предупреждение с количеством записей сохраняет требование подтверждённого удаления исторических данных.

**Альтернативы:** не сохранять результаты; хранить историю отдельными файлами; сохранять историю после удаления сайта; добавить отдельное удаление, архив или восстановление истории.

**Последствия:**

- экран проверки показывает историю от новой записи к старой и раскрывает технические подробности по запросу;
- подтверждение удаления сайта показывает число связанных записей и предупреждает об их необратимом удалении;
- отмена, GET, неверный токен и неизвестный сайт не изменяют сайт или историю;
- данные других сайтов сохраняются;
- содержимое страниц, снимки и параметры обхода в историю не входят;
- решение уточняет DEC-018 только для истории ручных проверок доступности.

**Условия пересмотра:** перед добавлением истории полного обхода, архивирования, восстановления или самостоятельного удаления истории принять отдельное решение о сроках хранения и связях данных.

### DEC-020. Хранение запусков обхода и удаление связанных данных сайта

**Дата:** 2026-07-15

**Статус:** принято

**Решение:** сохранять запуски обхода и упорядоченные метаданные обработанных страниц в SQLite до подтверждённого окончательного удаления сайта, без автоматического срока хранения. При таком удалении одной транзакцией удалять `AvailabilityCheck`, `CrawlPageRecord`, `CrawlRun` и `Site` выбранного сайта.

**Причина:** результаты обхода должны переживать перезапуск и оставаться целостно связанными с сайтом, а подтверждённое удаление не должно оставлять бесхозную историю. Отдельные точные количества проверок, запусков и записей страниц дают пользователю понятный объём необратимого действия.

**Альтернативы:** автоматический срок хранения; сохранение запусков после удаления сайта; отдельное удаление, архивирование или восстановление запусков; автоматическое каскадное удаление без явного предупреждения.

**Последствия:**

- сохраняются настройки запуска, итог, счётчики и упорядоченные метаданные страниц, но не HTML, текст страниц и найденные ссылки;
- отмена, GET, неверный токен и неизвестный сайт ничего не изменяют;
- данные другого сайта сохраняются;
- подтверждение удаления показывает отдельные количества проверок доступности, запусков обхода и записей страниц;
- отдельное удаление, архивирование и восстановление запусков не добавляются;
- решение уточняет DEC-018 и DEC-019 для данных обхода;
- будущие снимки этапа 3 этим решением не охватываются.

**Условия пересмотра:** перед добавлением автоматического срока хранения, самостоятельного управления запусками или снимков этапа 3 принять отдельное решение.

### DEC-021. Хранение снимков и переносимость слоя данных

**Дата:** 2026-07-16

**Статус:** принято

**Решение:** использовать существующий `CrawlRun` как контейнер снимка сайта. Для каждой успешно разобранной HTML-страницы добавлять одну связанную с `CrawlPageRecord` запись `CrawlPageSnapshot` со временем проверки в UTC, `Title`, `Description`, `H1`, нормализованным текстом, SHA-256 и полным списком внутренних ссылок.

`CrawlPageSnapshot` создаётся отдельной таблицей со связью один к одному с `CrawlPageRecord`. Найденные внутренние ссылки сохраняются как канонический UTF-8 JSON-текст. Сырой HTML, изображения, цены и другие ещё не согласованные данные не сохраняются.

При завершённом запуске публикуются снимки всех успешно разобранных HTML-страниц. При частично завершённом запуске сохраняются снимки успешных страниц вместе с существующими записями ошибок, но такой запуск не становится базой будущего сравнения. Базой сравнения считается последний завершённый запуск. Неуспешный, отложенный или прерванный запуск не публикует новый набор снимков и не заменяет предыдущую базу.

Записи страниц и их снимки публикуются при финализации запуска одной транзакцией. Ранее сохранённые запуски автоматически не дополняются снимками.

Снимки хранятся без автоматического срока хранения до подтверждённого окончательного удаления сайта. При удалении сайта одной транзакцией удаляются снимки страниц, записи страниц, запуски обхода, проверки доступности и сам сайт. Подтверждение показывает отдельное количество страниц с сохранённым содержимым. Самостоятельное удаление, архивирование и восстановление снимков пока не добавляются.

Текущая реализация остаётся на SQLite и SQLModel. PostgreSQL обязателен к моменту перехода продукта в многопользовательскую веб-версию, но не внедряется до соответствующего этапа. Переход на чистый SQLAlchemy выполняется только по отдельному решению при наличии измеримой объективной причины и не объединяется автоматически с переходом на PostgreSQL.

До будущей миграции новые решения слоя данных обязаны:

- использовать переносимые типы данных, явные внешние ключи, ограничения уникальности и транзакционные границы;
- не применять SQLite-специфичные JSON-функции, типы или SQL в доменной логике;
- изолировать создание engine, session и особенности блокировок в слое хранения;
- не связывать crawler и разбор HTML напрямую с SQLModel или конкретным драйвером базы данных;
- предусматривать явные и обратимые изменения схемы без потери ранее собранных данных.

Будущий переход на PostgreSQL выполняется отдельной задачей с резервной копией и проверяемым переносом данных. Приёмка миграции должна подтвердить количество сайтов, запусков, страниц и снимков, целостность связей, UTC-время, SHA-256 и найденные ссылки.

**Причина:** отдельная добавочная таблица минимально изменяет действующую схему, позволяет старой версии приложения игнорировать новые данные и даёт этапу 4 стабильную основу сравнения по запуску, URL и SHA-256. Переносимые типы и изоляция слоя хранения снижают стоимость обязательного будущего перехода на PostgreSQL без преждевременного усложнения локальной версии.

**Альтернативы:** добавить поля снимка прямо в `CrawlPageRecord`; хранить ссылки в SQLite-специфичном JSON; создать отдельную нормализованную таблицу каждой ссылки; сохранять сырой HTML; внедрить PostgreSQL или чистый SQLAlchemy уже на этапе 3; не сохранять частичные результаты.

**Последствия:**

- TASK-0014 может добавить одну новую таблицу и атомарное сохранение согласованных данных без изменения crawler и парсера под конкретную СУБД;
- сохранённый частичный запуск остаётся доступной историей, но не подменяет надёжную базу сравнения;
- канонический JSON-текст достаточно прост для текущего лимита обхода и переносится между SQLite и PostgreSQL без обязательной смены типа;
- при будущей необходимости поиска и анализа отдельных ссылок формат хранения может быть изменён отдельной миграцией;
- решение уточняет DEC-005, DEC-012, DEC-018, DEC-019 и DEC-020, не запуская переход на PostgreSQL или SQLAlchemy сейчас.

**Условия пересмотра:** автоматический срок хранения, отдельное управление снимками, значительное увеличение лимитов обхода, необходимость запросов по отдельным ссылкам, многопользовательская веб-версия, подтверждённые ограничения SQLite или измеримая причина перехода с SQLModel на чистый SQLAlchemy требуют отдельного решения и миграционного плана.

## 3. Назначение продукта

Marketing Intelligence — локальное веб-приложение для интернет-маркетолога. Оно должно наблюдать за собственными сайтами, сайтами конкурентов и доступными данными поисковой выдачи, сохранять историю изменений и помогать находить возможные причины изменения трафика и позиций.

Продукт должен превращать собранные данные в понятные пользователю:

- события;
- рассчитанные показатели;
- связи между событиями;
- гипотезы;
- рекомендации по дальнейшей проверке.

## 4. Основной пользователь

Основной пользователь первой версии — интернет-маркетолог без обязательных знаний программирования, системного администрирования, backend- и frontend-разработки.

Пользователь должен иметь возможность:

- установить и запустить приложение по понятной инструкции;
- выполнять основные действия через графический интерфейс;
- понимать состояние операций и сообщения об ошибках;
- получать полезный результат без изменения исходного кода;
- использовать основные функции без обязательных платных сервисов.

## 5. Границы продукта

### 5.1. Формат первой версии

- локальное веб-приложение;
- запуск на компьютере пользователя;
- работа через обычный настольный браузер;
- один локальный пользователь;
- локальное хранение основных данных;
- русский язык интерфейса;
- отсутствие обязательного платного сервера.

### 5.2. Состав MVP

MVP охватывает этапы 0–11 из `ROADMAP.md`:

1. подготовка проекта;
2. управление сайтами;
3. безопасный обход сайта;
4. сохранение снимков;
5. поиск изменений;
6. история изменений конкурентов;
7. карта структуры сайта;
8. локальный планировщик;
9. импорт данных Search Console и аналитики;
10. ограниченный мониторинг поисковой выдачи;
11. анализ возможных причин падения трафика;
12. системная проверка UX/UI.

### 5.3. Не входит в первую версию

- мобильное нативное приложение;
- публичная регистрация пользователей;
- оплата и подписки;
- агентская CRM;
- автоматическое изменение рекламных кампаний;
- генерация контента;
- автоматическая публикация изменений на сайтах;
- полноценная замена Ahrefs, Semrush или Screaming Frog;
- обязательная сложная AI-система;
- масштабирование на тысячи пользователей;
- обязательная облачная инфраструктура;
- многопользовательский режим;
- роли и права доступа;
- публичный API продукта.

#### 7.4.1. Извлечение и нормализация текста

При извлечении основного текста исключаются:

- навигационные и служебные блоки: шапка, футер и сайдбары;
- блоки комментариев, кнопки социальных сетей, формы обратной связи и подписки;
- рекламные баннеры, тизерные сети, блоки рекомендаций и другой динамический рекламный контент;
- скрипты, стили, всплывающие окна и технические атрибуты изображений;
- атрибуты изображений, кроме содержимого `alt`.

Нормализация выполняется в следующем порядке:

1. Unicode-нормализация NFC.
2. Unicode-совместимое приведение к нижнему регистру через `casefold()`.
3. Замена пунктуации и специальных символов пробелами, чтобы соседние слова не склеивались.
4. Свёртывание последовательностей пробельных символов до одного пробела.
5. Удаление пробелов только в начале и конце строки.

Нельзя удалять все пробелы через `replace(" ", "")`. Буква `ё` сохраняется и не заменяется на `е`.

#### 7.5.1. Классификация изменения

Алгоритм оценки изменения должен состоять из трёх уровней:

1. Расчёт базовой метрики сходства нормализованного текста.
2. Классификация изменения по уровню важности с учётом типа изменённого поля.
3. Необязательный семантический анализ на продвинутом уровне после подтверждения его необходимости и выбора бесплатного локального решения.

Точные числовые пороги и веса классов должны быть утверждены до реализации этапа 4.

#### 7.9.1. Приоритетные отчёты и поля импорта

**Google Search Console:**

1. `Pages.csv` / вкладка «Страницы» — критический приоритет. Обязательные поля: страница, клики (`Clicks`), показы (`Impressions`).
2. `Pages (Indexing).csv` / вкладка «Индексирование страниц» — высокий приоритет. Обязательные поля: URL, статус (`Status`), причина (`Reason`).

**Яндекс Метрика:**

1. Отчёт «Страницы входа» (`Landing Pages`) — критический приоритет.
2. Отчёт «Все страницы / Популярное» (`All Pages`) — высокий приоритет.
3. Обязательные поля первого варианта: адрес страницы, просмотры, отказы.

Точные названия столбцов могут меняться или локализоваться, поэтому импорт использует карту синонимов и обязательное подтверждение сопоставления пользователем.

В TASK-0035 реализован первый пункт: ручной импорт Google Search Console `Pages.csv` без дополнительных фильтров. Период вводится как две календарные даты. Клики и показы сохраняются точно, CTR рассчитывается из них, средняя позиция хранится без двоичной ошибки `float`. Предпросмотр ограничен временем и экземпляром приложения, исходный файл не сохраняется, а подтверждение записывает журнал и показатели одной транзакцией. Уникальность определяется сайтом, периодом и нормализованным URL; повторный импорт обновляет значения без автоматического удаления отсутствующих строк. Подробные правила приняты в DEC-032.

В TASK-0036 реализованы требования FR-086–FR-089: старые данные безопасно классифицируются, разделы `/competitors` и `/own-sites` изолированы, импорт Search Console перенесён в «Свои сайты», а защищённый перенос сохраняет идентификатор и связанные данные. Правила приняты в DEC-033. Автоматические интеграции, индексирование Search Console и отчёты Яндекс Метрики ещё не реализованы.

- микросервисы;
- Redis;
- Celery;
- Elasticsearch;
- Kubernetes;
- отдельное React-приложение;
- PostgreSQL;
- обязательный облачный сервер;
- обязательный OpenAI API или другой платный AI API.

Ограничение на PostgreSQL и обязательный сервер действует только до соответствующего подтверждённого этапа. По DEC-022 корпоративное многопользовательское развёртывание и PostgreSQL обязательны до начала рабочего использования готового продукта, но не внедряются заранее.

Добавление или замена технологии требует отдельного объяснения и фиксации решения в `DECISIONS.md`.
